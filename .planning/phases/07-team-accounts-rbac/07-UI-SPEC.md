## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge vertical padding |
| sm | 8px | Compact element spacing, button vertical padding |
| md | 16px | Default element spacing, card content padding |
| lg | 24px | Card padding (`p-6`), section padding |
| xl | 32px | Layout gaps, page vertical spacing |
| 2xl | 48px | Page `py-12` top padding |
| 3xl | 64px | Major page-level gaps (not used in Phase 7 surfaces) |

Exceptions:

- Role select dropdown within the invite row: 8px vertical padding (compact row form)
- Avatar initials circle: 32px diameter (width-8 height-8) — consistent with presence indicators in Phase 8
- Touch targets (buttons/icon buttons): minimum 32px × 32px per existing `min-w-[32px] min-h-[32px]` pattern from EditorToolbar

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 14px | 400 (normal) | 1.5 | Invite email field labels, member list rows, helper text, form field labels, panel section headers within modal |
| Heading | 16px | 600 (semibold) | 1.3 | Modal/panel title, member list section title, role confirmation heading |

Font sizes are sourced from the established codebase pattern:
`text-xs` = 12px, `text-sm` = 14px, `text-base` = 16px, `text-2xl` = 24px.
Exactly 2 weights across all Phase 7 surfaces: 400 (normal) for body and labels, 600 (semibold) for headings, display, and primary button labels.

## Color

| Role | Tailwind Class | Hex | Usage |
|------|---------------|-----|-------|
| Dominant (60%) | `bg-gray-50` / `bg-white` | #F9FAFB / #FFFFFF | Page background, modal interior, panel body |
| Secondary (30%) | `bg-gray-100` / `border-gray-200` | #F3F4F6 / #E5E7EB | Member list row hover, input borders, dividers, role select |
| Accent (10%) | `bg-blue-700` / `text-blue-700` | #1D4ED8 | Primary CTA only: "Send Invite" button, "Share" button in proposal header, focus ring (`ring-blue-500`) |
| Destructive | `bg-red-600` / `text-red-700` | #DC2626 / #B91C1C | Remove member action and its confirmation only |
| Status — shared | `bg-purple-50` / `text-purple-700` | #FAF5FF / #6D28D9 | Dashboard proposal status badge — team-owned proposals |

Accent reserved for:
1. "Share" button in proposal page header/toolbar
2. "Send Invite" primary button in the Share panel
3. Input focus rings (`focus:ring-2 focus:ring-blue-500 focus:border-blue-500`)
4. Role badge for `editor` role: `bg-blue-50 text-blue-700`

Secondary semantic colors (non-accent, non-destructive):
- `owner` role badge: `bg-gray-100 text-gray-700` (neutral — they already have full control)
- `viewer` role badge: `bg-gray-100 text-gray-500` (muted — read-only)
- `editor` role badge: `bg-blue-50 text-blue-700` (accent-derived — edit capability)
- Pending invite badge: `bg-yellow-50 text-yellow-700` (awaiting action state)
- Declined invite badge: `bg-red-50 text-red-700` (terminal error state)
- Dashboard "Shared" badge: `bg-purple-50 text-purple-700` (team-owned proposal, distinct from solo status badges)

## Interaction Contract

### Share Button placement

The Share button is added to the existing `flex items-start justify-between mb-6` row in `proposals/[id]/page.tsx`. It sits alongside the proposal title on the right side of that row. It is not rendered for `viewer` role users (server-side prop determines render).

### SharePanel open/close

- Open: clicking Share button sets `isPanelOpen = true` in client state
- Close: clicking the overlay backdrop, pressing Escape, or clicking the close (×) button
- Focus trap: on open, focus moves to the email input field
- On close: focus returns to the Share button

### Invite submit flow

1. User fills email + role, clicks Send Invite
2. Button enters loading state ("Sending...")
3. On success: email field clears, role resets to Editor, member list refreshes to show "Invited" badge for new entry
4. On error: inline error below email field appears; button returns to default state

### Role change flow

1. Owner clicks "Change role" text link on a member row
2. RoleChangeDropdown appears inline below that row
3. Clicking a role option: triggers PATCH, closes dropdown, badge updates optimistically
4. Clicking outside dropdown: closes without change

### Remove member flow

1. Owner clicks trash/× icon on a member row
2. Remove icon replaced inline with "Remove {email}? Keep member | Remove member"
3. "Keep member" restores the row to normal state
4. "Remove member" button: triggers DELETE, row is removed from list on success, seat count decrements

### Viewer restriction enforcement (client side)

- On proposal detail page: the entire Share button is not rendered for viewer-role users (server checks role before passing prop)
- Proposal editor: editor toolbar buttons remain active for `editor` role; for `viewer` role the ProposalEditor renders read-only (no mutation buttons visible)
- Note: server-side `requireProposalRole()` is the canonical enforcement — client-side suppression is UX convenience only

## Phase-Specific Notes

### New surfaces introduced in Phase 7

1. **Share button** — added to existing `proposals/[id]/page.tsx` header row
2. **SharePanel (modal)** — `src/components/team/SharePanel.tsx` — contains InviteForm + MemberList
3. **InviteForm** — within SharePanel, not a standalone page
4. **MemberList** — within SharePanel; member rows with role badges + owner actions
5. **RoleChangeDropdown** — inline dropdown within MemberList row
6. **RemoveMemberConfirmation** — inline replacement within MemberList row (no second modal)
7. **InviteAcceptPage** — `/invite/accept` route; auth-page shell pattern
8. **InviteDeclinePage** — `/invite/decline` route; auth-page shell pattern

### Dashboard impact

Proposals shared with a user via team membership must appear in the `/dashboard` "Recent Proposals" list. The existing proposal status badge system (`Ready`, `Processing`, `Draft`) is extended with a `Shared` badge using `bg-purple-50 text-purple-700` to distinguish team-owned proposals from solo ones. Badge copy: "Shared".

### No new nav items

Phase 7 adds no top-level nav or sidebar items. Team management is entirely contained within the proposal-level Share panel per D-07.

### Pending invite row in MemberList

Invites that have been sent but not yet accepted appear in the MemberList with:
- Avatar: initials circle with `bg-yellow-50 text-yellow-700` (pending state color signal)
- Email: text-sm text-gray-500 italic (pending — not yet a confirmed member)
- Role badge: `bg-yellow-50 text-yellow-700 text-xs font-semibold px-2 py-1 rounded-full` — "Invited"
- No change-role or remove actions for pending invites (owner cannot change role until accepted)
- Exception: owner can cancel a pending invite — trash icon shows, inline confirmation reads "Cancel invite to {email}?"
