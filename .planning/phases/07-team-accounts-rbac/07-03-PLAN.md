**2. Update `src/app/api/proposals/[id]/sections/route.ts`:**

In the GET handler, remove `.eq('user_id', user.id)` from the proposal_sections query. RLS handles access. Add `requireProposalRole` check:
```typescript
import { requireProposalRole } from '@/lib/auth/proposal-role'

// In GET:
const roleResult = await requireProposalRole(proposalId, 'viewer')
if (!roleResult) return NextResponse.json({ error: 'Forbidden' }, { status: 403 })

const { data, error } = await supabase
  .from('proposal_sections')
  .select('*')
  .eq('proposal_id', proposalId)
  .order('section_name', { ascending: true })
```

In the PATCH handler, enforce `editor` minimum role:
```typescript
// In PATCH:
const roleResult = await requireProposalRole(proposalId, 'editor')
if (!roleResult) return NextResponse.json({ error: 'Forbidden' }, { status: 403 })
```

Remove `.eq('user_id', user.id)` from the upsert — but keep `user_id: user.id` in the insert data (the column is NOT NULL). For team members editing, the `user_id` on the section should be the user who last edited it. Actually, `proposal_sections.user_id` was designed for solo use. For team context, the section belongs to the proposal not the user. But the column is NOT NULL with a FK. For now, keep setting `user_id: user.id` on upsert (the last editor). The RLS dual-policy on `proposal_sections` handles read access via proposal's team_id. For writes, RLS checks team membership.

Note: the existing RLS policy `"Users can manage own proposal_sections"` uses `(select auth.uid()) = user_id`. This means a team member who didn't create the section can't update it via RLS alone. The NEW team-access policy from migration 00005 allows team members to read/write sections of team proposals. So team members CAN write because the team-access policy grants it (Postgres OR logic across policies). Keep `user_id: user.id` in the upsert data for tracking who last edited.
